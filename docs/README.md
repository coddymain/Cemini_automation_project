## Можно ли перенести проект на другой продукт?

Да, архитектуру можно использовать как шаблон для другого web-приложения. Но перенос не означает просто поменять `BASE_URL`. У каждого продукта будут свои страницы, локаторы, авторизация, тестовые данные и API.

Что остается почти без изменений:
*   `pyproject.toml` и Poetry-подход;
*   `pytest.ini`;
*   базовая идея `conftest.py`;
*   Page Object Model;
*   фикстуры как Dependency Injection;
*   Pydantic Settings;
*   Allure, логи, скриншоты, видео и трейсы;
*   GitHub Actions pipeline;
*   правила типизации и структура тестов AAA.

Что нужно заменить под новый продукт:
*   `BASE_URL` и секреты в `.env` / GitHub Secrets;
*   page objects в папке `pages/`;
*   пользовательские сценарии в `tests/`;
*   способ логина и `authenticated_page`;
*   локаторы, URL страниц и тексты проверок;
*   `mock_server/`, если у продукта есть настоящий API или другой контракт;
*   тестовые данные и модели в `data/`.

## Чек-лист переноса на новый проект

1. Создать новый репозиторий или скопировать этот проект как основу.
2. Поменять имя проекта в `pyproject.toml`.
3. Заполнить `.env` под новый стенд.
4. Обновить GitHub Secrets: `BASE_URL`, логины, пароли и другие нужные значения.
5. Удалить или переписать Swag Labs page objects в `pages/`.
6. Написать первый `LoginPage` нового продукта.
7. Переписать `authenticated_page`, если авторизация устроена не через cookie `session-username`.
8. Добавить один самый простой smoke-тест.
9. Запустить `poetry run pytest -n 0`, чтобы отладить последовательно.
10. Включить обычный `poetry run pytest`, чтобы проверить параллельность.
11. Проверить Allure-отчет и артефакты падений.
12. Только после этого расширять покрытие.

## Минимальный сценарий для повторения

Новичку лучше не начинать с полного E2E. Первый переносимый сценарий должен быть маленьким:

1. Открыть главную страницу.
2. Проверить видимость одного стабильного элемента.
3. Авторизоваться.
4. Проверить URL или заголовок после входа.
5. Вынести локаторы в page object.
6. Передать page object через фикстуру.

Если этот путь работает локально и в CI, архитектура подключена правильно.

## Важные ограничения шаблона

*   Проект рассчитан на web UI. Для mobile, desktop или backend-only проекта структура потребует адаптации.
*   `authenticated_page` сейчас заточен под Swag Labs cookie. В реальном продукте чаще будет API-login, OAuth, JWT, SSO или storage state.
*   Локальный `mock_server/` — учебный инструмент. В рабочем проекте лучше использовать реальный тестовый API, контрактные моки или отдельный сервис в docker-compose.
*   `-n auto` ускоряет тесты, но требует независимости сценариев. Если тесты используют общую учетную запись и меняют одно и то же состояние, их нужно изолировать.

## Главное правило

Проект можно повторить неопытному тестировщику, если он идет по документации последовательно и меняет продуктовые детали маленькими шагами. Не нужно переписывать все сразу: сначала один стабильный тест, потом page object, потом фикстуры, потом отчеты и CI.
