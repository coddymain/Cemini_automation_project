# Шпаргалка 22: Масштабирование и параллельный запуск (pytest-xdist)

Поздравляем! Архитектура выстроена, тесты стабильны. Последний шаг к Enterprise-уровню — **скорость**.

## 1. Зачем нужен параллельный запуск?
UI-тесты (с открытием реального браузера) — это самый медленный вид тестирования. 
Один E2E тест может идти 5-10 секунд. Если у вас 500 тестов, последовательный запуск займет около часа.
Параллельный запуск разбивает эти 500 тестов на несколько потоков (воркеров), и время выполнения сокращается в несколько раз.

## 2. Инструмент: pytest-xdist
Для Pytest существует мощный плагин `pytest-xdist`. Он позволяет раскидать тесты по доступным логическим ядрам процессора вашего компьютера (или виртуальной машины в CI/CD).

## 3. Как настроить?
Установка через Poetry:
```bash
poetry add --group dev pytest-xdist
```

В файле `pytest.ini` в блок `addopts` добавляется флаг `-n auto`:
*   `-n auto` — плагин сам определит количество ядер процессора и запустит оптимальное число потоков.
*   `-n 4` — жестко задать 4 потока (полезно для стабильности на слабых серверах).
*   `-n 0` — отключить параллельность и выполнить тесты последовательно. Это удобно для отладки.

Примеры запуска:
```bash
# Полный быстрый прогон, использует настройки из pytest.ini
poetry run pytest

# Последовательный режим для диагностики
poetry run pytest -n 0

# Точечный запуск одного файла
poetry run pytest tests/test_mock_api.py -n 0 -v -s
```

## 4. Ограничения и правила параллельных тестов
Главное правило архитектуры, готовой к масштабированию: **Тесты не должны зависеть друг от друга.**
*   Плохо: Тест А добавляет товар в корзину, а Тест Б проверяет корзину. В параллели они могут запуститься одновременно и сломать друг друга.
*   Хорошо (Как у нас): Тест А и Тест Б используют строго изолированные браузерные контексты (фишка Playwright) и никак не пересекаются в памяти.

## 5. Что важно для нашего проекта

### UI-тесты
Playwright создает отдельный браузерный контекст для каждого теста. Поэтому cookie, localStorage и состояние страницы не протекают между тестами. Это делает UI-тесты хорошо совместимыми с `xdist`.

### API-тесты и mock server
API-тест `tests/test_mock_api.py` использует локальный FastAPI-сервер. Сейчас сервер поднимается автоматически через фикстуру `mock_api_server`, а API-клиент создается через `mock_api_context`.

При росте количества API-тестов есть два Enterprise-варианта:
*   поднимать один общий mock-сервис заранее в CI/CD;
*   выделять отдельный порт каждому `xdist` worker, чтобы процессы не конфликтовали за `127.0.0.1:8000`.

### Allure-результаты
Все worker-процессы пишут результаты в `allure-results`. Это штатный сценарий для `allure-pytest`: после завершения прогона Allure собирает единый отчет из всех JSON-файлов.

## 6. Чек-лист теста, готового к параллельному запуску

*   Тест сам готовит свои данные.
*   Тест не рассчитывает, что другой тест уже выполнился.
*   Тест не использует общие изменяемые файлы без уникальных имен.
*   Тест не зависит от порядка выполнения.
*   UI-тест получает `page` или page object через фикстуры.
*   API-тест получает клиент через фикстуру, а не создает глобальный клиент на уровне модуля.
*   Все секреты и URL берутся из `core/settings.py`, а не через `os.getenv`.

Теперь ваши пайплайны будут проходить максимально быстро!
