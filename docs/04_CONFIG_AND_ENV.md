# Шпаргалка 4: Управление данными и Конфигурация (Pydantic & .env)

В Enterprise-автоматизации строго запрещено "хардкодить" (писать напрямую в коде тестов) любые чувствительные или изменяемые данные: URL-адреса, логины, пароли, ключи API. 

## 1. Файл .env (Хранилище секретов)
Для хранения таких данных мы создали файл `.env` в корне проекта. 
**Важное правило:** Файл `.env` ВСЕГДА должен быть добавлен в `.gitignore` (что мы и сделали ранее). Он никогда не должен попасть в систему контроля версий (Git).

В нашем файле `.env` мы храним:
*   `BASE_URL` — базовый адрес сайта.
*   Логины и пароли тестовых пользователей.

## 2. Pydantic Settings (Строгая типизация конфигурации)

В старых проектах для чтения файла `.env` часто использовали встроенную библиотеку `os`:
```python
# Как НЕ надо делать:
import os
base_url = os.getenv("BASE_URL") # Если переменной нет, вернет None. Тест упадет позже с непонятной ошибкой.
```

Вместо этого мы используем современный подход — **Pydantic Settings**.
Мы создали файл `core/settings.py` с классом `Settings`.

### Почему этот подход лучше?
1.  **Строгая типизация:** Мы явно указываем `BASE_URL: str`. Если кто-то случайно запишет туда число или забудет добавить переменную в `.env`, Pydantic выдаст четкую ошибку `ValidationError` **еще до запуска тестов**.
2.  **Автоматическое чтение:** Нам не нужно писать код для чтения файла. Класс `BaseSettings` (от которого мы наследуемся) сам находит файл `.env` благодаря настройке `model_config = SettingsConfigDict(env_file=".env")`.
3.  **Автодополнение (IntelliSense):** Когда вы будете писать в тесте `config.`, редактор (VS Code) автоматически покажет вам все доступные переменные (BASE_URL, STANDARD_USER и т.д.), потому что они жестко прописаны в классе.

## Итог работы
Теперь в любом файле проекта (в страницах или тестах) мы можем безопасно получить конфигурацию всего одной строчкой:
```python
from core.settings import config

print(config.BASE_URL) # Выведет: https://www.saucedemo.com/
```
