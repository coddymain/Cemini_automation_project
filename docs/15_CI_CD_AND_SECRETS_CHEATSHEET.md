# Шпаргалка 15: Полный цикл CI/CD, Poetry в облаке и GitHub Secrets

Эта шпаргалка описывает полный цикл интеграции автотестов с GitHub Actions, включая решение типичных проблем при переносе локального кода на удаленный сервер.

## 1. Повседневный цикл Git (Важные нюансы)

Мы используем три основные команды для фиксации кода.
**ВНИМАНИЕ:** Крайне важно, из какой папки в терминале вы запускаете эти команды!

### Правильная последовательность работы:

1. **Возврат в корень проекта:**
   Всегда убеждайтесь, что терминал находится в главной папке проекта (корневой), иначе `git add .` не увидит файлы в папках выше уровнем.
   *Команда:* `cd ../..` (возвращает на уровень вверх. Используйте `cd`, пока не окажетесь в корне).

2. **Проверка статуса:**
   Смотрим, какие файлы изменены.
   *Команда:* `git status`

3. **Добавление в индекс (Stage):**
   Добавляем все изменения. Точка `.` означает "добавить всё в текущей директории и поддиректориях".
   *Команда:* `git add .`
   *(Если нужно добавить один файл: `git add путь/к/файлу.py`)*

4. **Создание коммита (Снимка):**
   Фиксируем изменения с понятным сообщением. Флаг `-m` позволяет написать текст.
   *Команда:* `git commit -m "Понятное описание того, что сделали"`

5. **Отправка на сервер:**
   Отправляем зафиксированные снимки на GitHub.
   *Команда:* `git push`

## 2. GitHub Actions: Структура и логика

GitHub Actions — это бесплатные серверы, которые запускают наш код. Сценарии для них хранятся строго в папке `.github/workflows/` в формате `.yml`.

**Ключевые блоки файла `playwright.yml`:**
*   `on:` — Когда запускать (например, `push` в ветку `main`).
*   `runs-on: ubuntu-latest` — На какой ОС запускать (используем Linux).
*   `steps:` — Пошаговая инструкция для сервера.

### Решение проблемы: Ошибка Poetry "package mode"
Если в логах Actions (Step: `poetry install`) вы видите ошибку:
`Error: The current project could not be installed: No file/folder found for package`

**Причина:** По умолчанию Poetry думает, что мы создаем библиотеку (пакет) для публикации в интернете, и ищет папку с исходным кодом пакета. Мы же просто используем Poetry для скачивания зависимостей (pytest, playwright).

**Решение:** Отключить `package-mode` в файле `pyproject.toml`.
Нужно добавить в конец `pyproject.toml` следующий блок:
```toml
[tool.poetry]
package-mode = false
```

## 3. GitHub Secrets: Безопасная передача паролей

В Enterprise-разработке файл `.env`, содержащий пароли и токены, **НИКОГДА** не отправляется на GitHub (он должен быть прописан в `.gitignore`).

Из-за этого виртуальный сервер GitHub Actions при запуске тестов не находит переменные и выдает ошибку (например, `pydantic_core.ValidationError: Field required [type=missing]`).

### Как безопасно передать переменные на сервер:

#### Шаг 1. Сохраняем переменные в сейф GitHub
1. Зайдите на страницу репозитория на GitHub.
2. Перейдите во вкладку **Settings** (Настройки).
3. В левом меню: **Security** -> **Secrets and variables** -> **Actions**.
4. Нажмите зеленую кнопку **"New repository secret"**.
5. Добавьте переменные из вашего локального `.env` по одной:
   *   Name: `BASE_URL` | Secret: `https://...`
   *   Name: `STANDARD_USER` | Secret: `standard_user`
   (И так далее для всех нужных переменных).

#### Шаг 2. Передаем секреты скрипту в файле .yml
Сервер сам по себе не подставит секреты. Ему нужно явно приказать вытащить их из сейфа прямо перед запуском тестов.
Для этого в файле `.github/workflows/playwright.yml` в шаг запуска тестов добавляется блок `env:`:

```yaml
    - name: Шаг 6. Запуск автотестов
      env:
        # Передаем переменные из секретов GitHub в окружение виртуальной машины
        BASE_URL: ${{ secrets.BASE_URL }}
        STANDARD_USER: ${{ secrets.STANDARD_USER }}
        LOCKED_OUT_USER: ${{ secrets.LOCKED_OUT_USER }}
        PROBLEM_USER: ${{ secrets.PROBLEM_USER }}
        VALID_PASSWORD: ${{ secrets.VALID_PASSWORD }}
      run: poetry run pytest -v
```
Синтаксис `${{ secrets.ИМЯ }}` — это системная команда GitHub, которая достает значение из вашего настроенного сейфа.
