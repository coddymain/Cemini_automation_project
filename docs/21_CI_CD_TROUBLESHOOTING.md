# Шпаргалка 21: Отладка CI/CD и проблема "Invalid cookie fields"

При переносе локально работающих автотестов в облачные CI/CD-среды (например, GitHub Actions) часто возникают специфические ошибки. Headless-браузеры (без графического интерфейса), запущенные на Linux-серверах, обладают более строгими политиками безопасности и проверки форматов данных.

## Описание проблемы
Во время попытки обойти UI-логин путем прямой установки Cookie (`page.context.add_cookies()`) выполнение прерывается с низкоуровневой ошибкой браузера:
`playwright._impl._errors.Error: BrowserContext.add_cookies: Protocol error (Storage.setCookies): Invalid cookie fields`

## Причины и Enterprise-решения

### 1. Блокировка на пустой странице (`about:blank`)
**Проблема:** По умолчанию Playwright открывает новую вкладку на системном адресе `about:blank`. Попытка вклеить Cookie для стороннего домена (например, `www.saucedemo.com`) на этой странице расценивается строгими политиками браузера как нарушение междоменной безопасности.
**Решение:** Сделать `page.goto(config.BASE_URL)` **до** установки Cookie, чтобы перевести контекст на нужный домен.

### 2. Динамический парсинг домена и URL
**Проблема:** Использование готового параметра `"url": config.BASE_URL` при добавлении Cookie иногда отвергается протоколом CDP (Chrome DevTools Protocol) в Linux-контейнерах из-за строгих проверок формата. При этом жестко хардкодить домен (например, `"www.saucedemo.com"`) — это плохая практика, так как при смене окружения (например, переезд на тестовый стенд) придется переписывать код фикстур.
**Решение:** Динамически извлекать чистый домен из `BASE_URL` с помощью встроенной библиотеки `urllib.parse` и передавать низкоуровневую связку `domain` и `path`.

```python
from urllib.parse import urlparse
parsed_domain = urlparse(config.BASE_URL).hostname

# Затем используем parsed_domain в параметре "domain" для Cookie
```

### 3. Невидимые символы в GitHub Secrets (Самая коварная причина)
**Проблема:** При копировании значений логина или пароля в интерфейс GitHub Secrets в конец строки часто случайно попадает невидимый символ переноса строки (`\n` или `\r\n`). 
Когда Playwright читает такой секрет (например, `"standard_user\n"`) и пытается установить его в качестве значения Cookie, Chromium отклоняет его, так как спецификация HTTP запрещает перенос строки в теле Cookie.

**Решение:** Всегда обрабатывайте строковые секреты из переменных окружения в CI/CD методом `.strip()` в Python перед использованием их в чувствительных операциях!
```python
# Надежное получение значения без невидимого мусора:
cookie_value = config.STANDARD_USER.strip()
```

## Итог
Отладка CI/CD требует максимальной изоляции компонентов. Добавление подробного логирования (`logger.info`) каждого шага и анализ сырых значений переменных (как в случае с поиском `\n` в логах) — это главный инструмент Enterprise-автоматизатора для борьбы с плавающими (flaky) инфраструктурными ошибками.
