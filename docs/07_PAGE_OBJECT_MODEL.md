# Шпаргалка 7: Архитектура Page Object Model (POM)

При переходе от простых скриптов к Enterprise-автоматизации мы обязаны использовать паттерн **Page Object Model (POM)**. 

## 1. Проблема тестов без POM ("Простыни кода")
Если писать тесты как в нашем старом файле `test_login_no_pom.py`, логика самого теста (проверки, assert-ы) смешивается с технической реализацией (поиск элементов, клики, CSS-селекторы). 

**Почему так делать нельзя в реальных проектах:**
1. **Дублирование:** Селектор поля `username` будет скопирован в десятках тестов.
2. **Сложность поддержки (Maintenance Nightmare):** Если разработчики изменят дизайн сайта и поменяют селекторы, нам придется вручную искать и править их во всех тестовых файлах.
3. **Плохая читаемость:** Тест становится похож на низкоуровневую инструкцию для робота, а не на понятный бизнес-сценарий.

---

## 2. Решение: Паттерн POM
**Суть паттерна:** Мы выносим работу с каждой конкретной веб-страницей (или её логическим блоком) в отдельный Python-класс. 
Тесты больше не ищут элементы напрямую. Они общаются со страницей через понятные методы.

### Шаг 1: BasePage (Базовый класс)
**Файл:** `pages/base_page.py`

Это фундамент (родительский класс) для всех будущих страниц. Зачем он нужен? Чтобы не писать один и тот же код в каждом классе страницы.

```python
from playwright.sync_api import Page

class BasePage:
    def __init__(self, page: Page):
        self.page = page
```
*   `__init__` — это конструктор класса в Python. Он вызывается при создании объекта.
*   `self.page = page` — мы сохраняем объект браузера (вкладку), который нам передал тест, внутрь нашего класса, чтобы другие методы могли с ним работать.

### Шаг 2: LoginPage (Наследник BasePage)
**Файл:** `pages/login_page.py`

Этот класс "знает" всё про страницу авторизации. Давайте разберем его **построчно**, чтобы понимать каждую деталь механики. Нажмите на строку кода, чтобы перейти к её описанию, и на стрелочку в описании, чтобы вернуться к коду.

```python
from pages.base_page import BasePage

class LoginPage(BasePage):
    
    # --- БЛОК 1: ПРИВАТНЫЕ ЛОКАТОРЫ ---
    @property
    def _username_input(self):
        return self.page.locator('[data-test="username"]')

    @property
    def _password_input(self):
        return self.page.locator('[data-test="password"]')

    @property
    def _login_button(self):
        return self.page.locator('[data-test="login-button"]')

    # --- БЛОК 2: ПУБЛИЧНЫЕ ЛОКАТОРЫ ---
    @property
    def error_message(self):
        return self.page.locator('[data-test="error"]')

    # --- БЛОК 3: БИЗНЕС-ЛОГИКА (МЕТОДЫ) ---
    def login(self, username, password):
        self._username_input.fill(username)
        self._password_input.fill(password)
        self._login_button.click()
```

#### Построчный разбор (Что, откуда, почему и как):

*   **`from pages.base_page import BasePage`**
    *   *Что это:* Импорт родительского класса.
    *   *Откуда:* Из соседнего файла `base_page.py` в папке `pages`.
    *   *Зачем:* Чтобы не дублировать код инициализации браузера (`__init__`), мы берем его из `BasePage`.

*   **`class LoginPage(BasePage):`**
    *   *Что это:* Объявление класса страницы логина. В скобках указано `(BasePage)`.
    *   *Как работает:* Это механизм **наследования**. `LoginPage` получает все свойства и методы `BasePage`. Когда мы создадим объект `LoginPage(page)`, он автоматически вызовет `__init__` из `BasePage` и сохранит `self.page`.

*   **`@property`**
    *   *Что это:* Встроенный декоратор Python.
    *   *Зачем:* Превращает метод (функцию) в "свойство" (атрибут). 
    *   *Как работает:* Без `@property` нам пришлось бы писать вызов метода со скобками: `self._username_input().fill(...)`. С `@property` мы обращаемся к нему как к переменной: `self._username_input.fill(...)`. Код выглядит чище и лаконичнее.

*   **`def _username_input(self):`**
    *   *Что это:* Метод, возвращающий локатор (адрес) элемента на странице.
    *   *Почему тут нижнее подчеркивание `_`:* Это **Инкапсуляция**. В Python нет жесткого слова `private`. Подчеркивание — это договор (конвенция) между программистами. Оно означает: *"Этот локатор нужен только для внутренних нужд класса LoginPage. Не вызывай его из тестов!"*.
    *   *Как работает:* Когда метод `login` запрашивает `self._username_input`, эта функция выполняется и возвращает объект локатора Playwright.

*   **`return self.page.locator('[data-test="username"]')`**
    *   *Что это:* Прямое обращение к Playwright для поиска элемента.
    *   *Откуда:* `self.page` пришел из `BasePage`. Метод `.locator(...)` — это встроенная функция Playwright.
    *   *Как работает:* Playwright запоминает CSS-селектор `[data-test="username"]`, но **не ищет** его на странице прямо сейчас (это называется ленивый поиск, "lazy evaluation"). Он пойдет искать его только тогда, когда мы попытаемся с ним что-то сделать (например, `.fill()`).

*   **`def error_message(self):`**
    *   *Что это:* Тоже локатор, но **публичный** (без `_` в начале).
    *   *Почему:* Потому что текст ошибки нам нужно проверять в самом тесте (внутри файла `test_login.py` мы напишем `expect(login_page.error_message).to_have_text(...)`). Раз тест имеет право обращаться к этому локатору напрямую, мы не ставим подчеркивание.

*   **`def login(self, username, password):`**
    *   *Что это:* Публичный метод, описывающий **бизнес-действие** ("Авторизоваться").
    *   *Зачем:* Тест не должен знать, как именно заполняются поля. Он просто командует: "Авторизуй пользователя X с паролем Y".
    *   *Как работает:* 
        1. `self._username_input.fill(username)` — обращается к приватному свойству локатора и говорит Playwright "введи текст `username` в этот элемент".
        2. Аналогично для пароля.
        3. `self._login_button.click()` — нажимает на кнопку.

### 💡 Итог: Зачем нужна Инкапсуляция?

**Инкапсуляция** спасает от хрупкости тестов:
1.  **Тесты (`test_login.py`)** — это пользователи. Они знают только бизнес-логику: *"Авторизуй меня"*. Они не знают, что под капотом есть какие-то CSS-селекторы.
2.  **Страница (`LoginPage`)** — это капсула. Она скрывает в себе всю грязную работу (локаторы, клики, ожидания) за закрытыми дверями (через `_`).

Если верстальщики изменят кнопку "Login", сломается только одна строчка внутри `_login_button` в файле `login_page.py`. Все 100 тестов, которые используют метод `login()`, продолжат работать как ни в чем не бывало.

### Шаг 3: InventoryPage 
**Файл:** `pages/inventory_page.py`

Описывает страницу товаров (куда мы попадаем после логина). Именно здесь мы недавно исправляли ошибку парсинга некорректного локатора `{span.title }`.

```python
from pages.base_page import BasePage

class InventoryPage(BasePage):
    @property
    def title(self):
        # Исправленный, валидный CSS-селектор: тег span с классом title
        return self.page.locator('span.title')
```

## 3. Как выглядят тесты с POM
**Файл:** `tests/test_login.py`

Посмотрите, насколько чистым стал код теста. Теперь это прямое описание шагов пользователя:

```python
from playwright.sync_api import Page, expect
from core.settings import config
from pages.login_page import LoginPage
from pages.inventory_page import InventoryPage

def test_successful_login(page: Page):
    # 1. Arrange (Инициализируем объекты страниц)
    login_page = LoginPage(page)
    inventory_page = InventoryPage(page)
    
    # 2. Act (Выполняем шаги пользователя)
    login_page.open(config.BASE_URL)
    login_page.login(config.STANDARD_USER, config.VALID_PASSWORD)
    
    # 3. Assert (Проверки: ожидаем нужный URL и текст на новой странице)
    expect(page).to_have_url(f"{config.BASE_URL}inventory.html")
    expect(inventory_page.title).to_have_text("Products")
```

**Главный итог (Почему POM это круто):** 
Если завтра кнопка логина изменит свой `data-test` атрибут, наши тесты *не сломаются*. Мы зайдем в **один единственный файл** `login_page.py`, поменяем локатор там, и все 50 тестов, которые используют логин, снова заработают!

## 4. Запуск тестов через Poetry

Поскольку мы используем **Poetry** для изоляции проекта (как описано в шпаргалке `03_POETRY_AND_DEPS.md`), мы запускаем наш обновленный тест строго через виртуальное окружение Poetry:

```bash
poetry run pytest tests/test_login.py -v
```
*   `poetry run` — гарантирует, что pytest и Playwright берутся из нашего изолированного окружения `.venv`, а не из глобальной системы macOS.
*   `-v` — флаг verbose для подробного вывода (чтобы видеть название прошедшего теста).
