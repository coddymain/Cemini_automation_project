# Шпаргалка 14: Управление версиями (Git) и CI/CD (GitHub Actions)

В Enterprise-разработке код никогда не хранится только локально, а тесты не запускаются только вручную. Мы используем систему контроля версий (Git) для отслеживания истории изменений, удаленные репозитории (GitHub) для хранения и командной работы, а также CI/CD для автоматического запуска проверок.

## 1. Локальная работа с Git (Повседневный цикл)

### Шаг 1. Проверка статуса
Прежде чем сохранять изменения, мы всегда проверяем, какие файлы были изменены или добавлены:
```bash
git status
```
*   `git status` — показывает текущее состояние рабочего дерева: какие файлы изменены, какие новые (untracked), какие подготовлены к коммиту.

### Шаг 2. Добавление файлов в индекс (Staging)
Мы выбираем файлы, которые хотим включить в следующий снимок (коммит):
```bash
git add .
```
*   `git add` — добавляет изменения в индекс (подготавливает к сохранению).
*   `.` (точка) — означает "добавить все измененные и новые файлы в текущей директории и поддиректориях", игнорируя то, что указано в `.gitignore`.

### Шаг 3. Создание коммита (Снимка)
Мы фиксируем изменения с понятным сообщением:
```bash
git commit -m "Текст сообщения"
```
*   `git commit` — создает коммит (снимок состояния файлов).
*   `-m` — флаг (message), который позволяет передать текст описания прямо в терминале.

## 2. Интеграция с GitHub (Удаленный репозиторий)

### Шаг 4. Проверка подключения к GitHub
Чтобы убедиться, что наш локальный проект связан с удаленным сервером:
```bash
git remote -v
```
*   `git remote` — команда для работы с удаленными репозиториями.
*   `-v` (verbose) — флаг "подробно", выводит не только имена (например, `origin`), но и сами URL-адреса для скачивания (fetch) и отправки (push).

### Шаг 5. Привязка репозитория (Делается один раз)
Если репозиторий еще не связан:
```bash
git remote add origin <ССЫЛКА_ИЗ_GITHUB>
```
*   `git remote add` — добавляет новую связь.
*   `origin` — стандартное имя (псевдоним) для основного удаленного репозитория.

### Шаг 6. Переименование главной ветки (Делается один раз)
Современный стандарт для названия главной ветки — `main`:
```bash
git branch -M main
```
*   `git branch` — команда управления ветками.
*   `-M` — флаг для принудительного переименования текущей ветки.

### Шаг 7. Отправка кода на сервер (Push)
После того как коммиты сделаны локально, их нужно отправить на GitHub:
```bash
git push -u origin main
```
*   `git push` — команда отправки локальных коммитов в удаленный репозиторий.
*   `-u` (или `--set-upstream`) — флаг, который связывает локальную ветку с удаленной. В будущем достаточно будет писать просто `git push`.
*   `origin` — имя удаленного репозитория.
*   `main` — имя ветки.

## 3. Настройка CI/CD (GitHub Actions)

CI/CD (Continuous Integration / Continuous Deployment) — это практика автоматического запуска скриптов (в нашем случае — тестов) при каждом обновлении кода на сервере.

Для GitHub Actions инструкции (рабочие процессы или workflows) хранятся в `.yaml` файлах внутри специальной папки: `.github/workflows/`.

**Основные блоки конфигурации `.yml` файла:**
*   `on:` — Триггеры (когда запускать сценарий, например, при `push` в ветку `main`).
*   `jobs:` — Задачи (что именно делать).
*   `runs-on: ubuntu-latest` — Выбор операционной системы виртуальной машины (сервера), на которой будет выполняться работа.
*   `steps:` — Пошаговые инструкции (скачать код, установить программы, запустить скрипты).
*   `uses:` — Вызов готового чужого "экшена" (например, `actions/checkout@v4` для скачивания кода на виртуальную машину).
*   `run:` — Выполнение консольной команды Linux на виртуальной машине (например, `poetry run pytest`).
