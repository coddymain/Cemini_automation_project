# Шпаргалка 13: Работа с Данными, Pydantic Моделями и Настройками (.env)

В этой шпаргалке мы разберем, как мы организовали хранение тестовых данных и настроек проекта, чтобы код был чистым, надежным и с автодополнением в редакторе.

## 1. Зависимости (Что мы установили)
Для работы с моделями данных и настройками мы используют библиотеку **Pydantic**. 
В файле `pyproject.toml` (или с помощью команды `poetry add`) были добавлены:
*   `pydantic` — мощная библиотека для проверки данных (валидации) и управления ими с помощью аннотаций типов Python.
*   `pydantic-settings` — расширение для Pydantic, которое умеет автоматически читать переменные из файла `.env` и превращать их в удобные Python-объекты.

## 2. Папка `core` и файл `settings.py` (Настройки окружения)
Мы создали папку `core/`, где хранятся базовые, фундаментальные механизмы проекта.
Файл `core/settings.py` отвечает за загрузку всех секретов (паролей, логинов, базового URL) из скрытого файла `.env`.

**Как это работает (разбор кода `core/settings.py`):**
```python
from pydantic_settings import BaseSettings, SettingsConfigDict

class Settings(BaseSettings):
    # 1. Мы описываем, какие переменные мы ожидаем увидеть в файле .env.
    # Двоеточие (:) и 'str' означают, что мы жестко требуем, чтобы это была строка.
    BASE_URL: str
    STANDARD_USER: str
    VALID_PASSWORD: str

    # 2. Настройки самого Pydantic
    model_config = SettingsConfigDict(
        env_file=".env",              # Указываем, из какого файла брать значения
        env_file_encoding="utf-8",    # Кодировка файла (чтобы не было проблем с символами)
        extra="ignore"                # Если в .env есть "лишние" переменные, которых нет в классе, Pydantic не упадет с ошибкой, а просто проигнорирует их.
    )

# 3. Создаем "глобальный" объект config. 
# В момент вызова Settings() библиотека пойдет в файл .env, прочитает его и заполнит переменные.
config = Settings()
```

**Зачем это нужно?**
Теперь в любом месте (например, в фикстурах или тестах) мы можем написать `from core.settings import config` и безопасно использовать `config.BASE_URL`. Если в `.env` файле забудут указать URL, проект упадет сразу при запуске, а не где-то в середине теста с непонятным "NoneType error".

## 3. Папка `data` и файл `models.py` (Тестовые данные)
Мы вынесли генерацию и хранение тестовых данных в отдельную папку `data/`. Это отделяет "информацию" (данные) от "логики работы со страницами" (которая лежит в `pages/`).

Файл `data/models.py` содержит классы данных (Модели).

**Разбор кода `data/models.py`:**
```python
from pydantic import BaseModel

# 1. Создаем "Чертеж" (Схему) нашего покупателя.
# Наследуемся от BaseModel из Pydantic. Это дает классу супер-способности по проверке типов.
class Customer(BaseModel):
    first_name: str
    last_name: str
    postal_code: str

# 2. Создаем конкретный экземпляр покупателя (готовые данные) по этому чертежу.
test_customer = Customer(first_name="Ivan", last_name="Ivanov", postal_code="123456")
```

## 4. Как мы использовали данные в тестах (Практика)

Давайте проследим путь от "чертежа" до использования в самом тесте. Это состоит из двух шагов:

### Шаг 1. Настройка Page Object (`pages/checkout_page.py`)
Мы учим метод страницы принимать **не отдельные строки**, а целый объект покупателя.

**Что и где прописано:**
```python
# Файл: pages/checkout_page.py

# Импортируем только сам "чертеж" (класс Customer), чтобы указать тип (type hint)
from data.models import Customer 

class CheckoutPage:
    # ... локаторы ...

    # Указываем, что аргумент customer должен быть строго типа Customer
    def fill_personal_info(self, customer: Customer) -> None:
        
        # КАК ЭТО РАБОТАЕТ:
        # Когда мы печатаем 'customer.', редактор сразу подсказывает нам доступные поля:
        # first_name, last_name, postal_code. 
        # Мы берем данные из объекта и передаем их в метод .fill() локатора Playwright.
        self.first_name_input.fill(customer.first_name)
        self.last_name_input.fill(customer.last_name)
        self.postal_code_input.fill(customer.postal_code)
```

### Шаг 2. Использование в Тесте (`tests/test_e2e.py`)
В самом тесте мы передаем готового "покупателя" в метод страницы.

**Что и где прописано:**
```python
# Файл: tests/test_e2e.py

# Импортируем готовый объект с тестовыми данными (экземпляр покупателя)
from data.models import test_customer

def test_full_checkout_flow(page: Page, checkout_page: CheckoutPage, ...):
    # ... предыдущие шаги теста ...
    
    # 4. Оформление заказа: Шаг 1
    # Мы передаем весь объект test_customer в метод одной переменной!
    checkout_page.fill_personal_info(test_customer)
    
    # ... продолжение теста ...
```

### Почему такой подход — это круто?
1.  **Чистота в тесте:** Вместо `checkout_page.fill_personal_info("Ivan", "Ivanov", "123456")` мы пишем коротко и понятно: `fill_personal_info(test_customer)`.
2.  **Защита от ошибок:** Если метод страницы ждет `Customer`, а вы передадите туда строку "Ivan", ваш редактор кода (или тайпчекер) сразу подсветит это красным, еще до запуска теста.
3.  **Автодополнение (IntelliSense):** Внутри `checkout_page.py` редактор знает, что такое `customer`, и сам предлагает написать `customer.first_name`. Не нужно вспоминать, как именно называется поле.
4.  **Удобство масштабирования:** Если форма на сайте изменится, и добавится поле "Номер телефона", нам нужно будет добавить его **только в одном месте** — в чертеж `Customer` в файле `models.py`. Метод `fill_personal_info` просто начнет брать новое поле `customer.phone_number`, а сам вызов в тесте `test_e2e.py` вообще не придется менять!
